# RefactoringLab
A refactoring exercise project.

I first heard about this challenge in 2023. I updated the original project from .NET 5 to .NET 6.
The idea is to work through the proposed challenge described in `CHALLENGE.md`.

In June 2026, I finally decided to give it a go.

1. First, I started by analyzing the existing codebase. 

2. The `LegacyApp.Consumer` project did not include all the files and resources needed to run properly. Because of that, I decided to start with an integration test that works as a characterization test, covering the same scenario used in its `Program` file.

3. I created integration tests using Testcontainers, Docker, and SQL Server.

4. Once I had a characterization test in place, I moved on to thinking about unit tests.

5. I added a unit test project.

6. I changed the `LegacyApp` project to expose its internals to the unit test project.

7. I gradually made small changes, injecting the dependencies through a second internal constructor. This constructor is not visible to the `LegacyApp.Consumer` project.

8. The `UserService.AddUser()` method created and disposed of a `UserCreditServiceClient` instance directly with a `using` statement. I did not want to change the logic inside of `AddUser`, so I first moved the `new` statement to a method called `GetUserCreditService`.

   Then I noticed that the `UserCreditServiceClient` class has a `GeneratedCode` attribute. Because of that, I assumed it was generated by a tool and I should not edit that file.

   As is common with generated code, the tool had marked the class as partial, which allowed me to extend it in a separate file without editing the generated file directly. So I added a new file for the same class and extended it to implement a new interface: `IDisposableUserCreditService`. This interface combines `IDisposable` and `IUserCreditService`.

   After that, I changed `GetUserCreditService` to work as a factory for `IDisposableUserCreditService` instead of `UserCreditServiceClient`.

9. The last dependency I replaced was `System.DateTime.Now`. Since this project targets .NET 6, I chose a simple solution: provide the current local date and time through a DateTime variable. There is no need to use IClock or TimeProvider here.

10. At each step where I introduced dependency injection, I added unit tests to expose and preserve the existing behavior of `UserService.AddUser()`.

11. Once all dependencies were injectable, I refactored the class and the method. The logic was moved into functions. With the approach I used, these methods are now static pure functions. They can be moved to external services and reused. Also, if they grow in complexity, they can be tested directly and independently.

The refactoring is done. The class and method were refactored without impacting the `LegacyApp.Consumer` project: no new public methods, and no new transitive dependencies.

## Update

When I first found the source code for this challenge, it already depended on a vulnerable NuGet package.

I do not know whether this was a hidden part of the original challenge, but the challenge is from 2023, and since then many NuGet packages have been marked as vulnerable.

It does not make sense to keep referencing vulnerable packages.

So I will do here what I would do in a real code challenge: communicate the issue clearly. I will apply the fixes for the vulnerable packages in a separate pull request:
